# 微信小程序开发者工具 CLI MCP 设计方案

## 一、项目概述

### 1.1 目标

创建一个完整的 MCP (Model Context Protocol) 服务器，封装微信开发者工具 CLI 和 HTTP API 的所有功能，使 AI 助手能够通过自然语言调用小程序开发工具的各项能力。

### 1.2 功能范围

覆盖微信开发者工具的**全部**功能：
- 登录管理（CLI + HTTP）
- 代码预览与上传（CLI + HTTP）
- 项目窗口管理（CLI + HTTP）
- 云开发操作（CLI + HTTP）
- 自动化测试（CLI）
- 缓存管理（CLI + HTTP）
- npm 构建（CLI + HTTP）

**CLI vs HTTP 使用策略**:
- **优先使用 CLI**：大多数操作通过 CLI 更稳定
- **HTTP 补充场景**：需要更精细的缓存控制、批量操作、或需要异步回调的场景
- **功能重叠时**：默认使用 CLI，提供 `useHttp` 参数允许切换到 HTTP

### 2.2 全局参数映射

所有工具共享以下通用参数：

| 参数 | 类型 | 环境变量 | 说明 |
|------|------|----------|------|
| `project` | string | - | 项目路径 |
| `appid` | string | - | 小程序 AppID |
| `extAppid` | string | - | 第三方平台 AppID |
| `port` | number | `WEAPP_PORT` | **HTTP 服务端口**（重要：不是 auto 端口）|
| `lang` | string | `WEAPP_LANG` | 语言 (en/zh) |
| `debug` | boolean | `WEAPP_DEBUG` | 调试模式 |

**重要约束提醒**:
> `--port` 参数指定的是**HTTP 服务端口**，不是自动化端口。
> - 如果工具未启动，会以此端口启动 HTTP 服务
> - 如果工具已启动但端口不同，需要先执行 `weapp_quit` 再重新启动
> - HTTP 服务端口号可以在启动后的输出中解析，或从 `.ide` 文件中读取

### 3.2 HTTP 端口检测

HTTP 服务端口号存储在用户目录下的 `.ide` 文件中：

| 平台 | 文件路径 |
|------|----------|
| macOS | `~/Library/Application Support/微信开发者工具/<MD5>/Default/.ide` |
| Windows | `~/AppData/Local/微信开发者工具/User Data/<MD5>/Default/.ide` |

MD5 计算规则：`MD5(`${installPath}${nwVersion}`)`

### 3.4 CLI vs HTTP 选择策略

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 启动/关闭工具 | CLI | 更可靠，可以直接控制进程 |
| 开启自动化 | CLI | 需要传递 auto-port 等参数 |
| 预览/上传 | CLI (默认) / HTTP (可选) | CLI 更稳定，HTTP 适合批量操作 |
| 清除缓存 | HTTP | 支持更精细的缓存类型控制 |
| 云开发操作 | CLI (默认) / HTTP (可选) | 功能相同，按场景选择 |

### 4.2 核心模块设计

#### 4.2.1 CLI 客户端 (`cli-client.ts`)

负责与微信开发者工具 CLI 交互：

```typescript
class CLIClient {
  private cliPath: string;
  
  // 执行 CLI 命令
  async execute(command: string, args: string[], options?: ExecuteOptions): Promise<CLIResult>;
  
  // ========== 改造后的方法 ==========
  
  // 启动开发者工具（不带项目）
  async openTool(options: {
    httpPort?: number;      // HTTP 服务端口
    lang?: string;          // 语言
    debug?: boolean;        // 调试模式
  }): Promise<CLIResult & { httpPort?: number }>;
  
  // 打开指定项目
  async openProject(projectPath: string, options?: {
    httpPort?: number;
    lang?: string;
    debug?: boolean;
  }): Promise<CLIResult & { httpPort?: number }>;
  
  // 退出工具（带重试机制）
  async quit(retryCount?: number): Promise<CLIResult>;
  
  // 关闭项目（带重试机制）
  async close(projectPath?: string, retryCount?: number): Promise<CLIResult>;
}
```

**关键改造点**:
1. **区分启动工具/项目**: `openTool()` vs `openProject()`
2. **返回 HTTP 端口**: 从 CLI 输出中解析 HTTP server 端口号
3. **端口冲突检测**: 如果返回的端口与指定端口不同，明确提示用户
4. **实例唯一性保证**: 启动前检测并关闭其他实例

## 六、错误处理策略

### 6.1 实例冲突处理

当启动工具/项目时检测到其他实例：

```typescript
async function ensureSingleInstance(): Promise<void> {
  // 1. 检测是否有其他实例运行
  const status = await checkStatus();
  
  if (status.isRunning) {
    // 2. 尝试优雅关闭（多次重试）
    for (let i = 0; i < 3; i++) {
      await cliClient.quit();
      await sleep(2000);
      if (!(await checkStatus()).isRunning) break;
    }
    
    // 3. 如果仍然存在，提示用户手动处理
    if ((await checkStatus()).isRunning) {
      throw new Error("无法关闭现有实例，请手动关闭微信开发者工具后重试");
    }
  }
}
```

### 6.2 端口不匹配处理

当实际 HTTP 端口与指定端口不一致时：

```typescript
if (specifiedPort && actualPort !== specifiedPort) {
  return {
    content: [{
      type: "text",
      text: `⚠️ HTTP 端口不匹配\n\n指定的端口: ${specifiedPort}\n实际的端口: ${actualPort}\n\n可能原因:\n   1. 指定端口已被占用\n   2. 工具已在其他端口运行\n\n建议: 先执行 weapp_quit 关闭工具，然后重新启动。`
    }],
    isError: true
  };
}
```

## 七、配置方案

### 7.1 环境变量

| 变量名 | 说明 | 默认值 |
|--------|------|--------|
| `WEAPP_CLI_PATH` | CLI 工具路径 | 自动检测 |
| `WEAPP_PORT` | **HTTP 服务端口** | 自动分配 |
| `WEAPP_LANG` | 语言设置 | zh |
| `WEAPP_DEBUG` | 调试模式 | false |

### 7.2 重要约束

1. **HTTP 端口 vs 自动化端口**:
   - `--port` 参数指定的是 HTTP 服务端口（用于 CLI/HTTP 通信）
   - `--auto-port` 参数指定的是自动化端口（用于自动化测试）
   - 两者完全不同，不可混淆

2. **实例唯一性**:
   - 启动工具前会尝试关闭其他实例
   - 打开项目前会尝试关闭其他项目
   - 这是为了保证操作的可预测性

3. **端口检测延迟**:
   - HTTP 服务和自动化服务启动后可能有延迟
   - 需要等待并轮询检测端口是否真正被监听
